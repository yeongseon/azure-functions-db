# 테스트 전략

## 1. 목표

- event loss 없는지 검증
- duplicate는 허용하되 예상 범위 내인지 검증
- DB별 query correctness 보장
- restart / crash / lease race에서 semantics 유지

## 2. 테스트 레벨

### 2.1 Unit
대상:
- cursor comparator
- checkpoint serializer
- event_id generation
- SQL builder
- retry classifier

### 2.2 Contract
모든 adapter가 같은 입력/출력 contract를 만족하는지 검증.

필수 항목:
- ordering
- empty batch semantics
- checkpoint generation
- timezone normalization
- duplicate-safe cursor

### 2.3 Integration
실제 DB 컨테이너로 검증.
- PostgreSQL
- MySQL/MariaDB
- SQL Server

### 2.4 E2E
Azure Functions local runtime + Azurite + DB container 조합.

## 3. 핵심 시나리오

### S1 첫 실행
초기 checkpoint 없음 -> 첫 batch 처리 성공

### S2 동일 timestamp 100건
동일 `updated_at` row 다수 -> 누락 없이 순차 처리

### S3 handler 실패
checkpoint 미전진 확인

### S4 commit 실패
중복 가능하지만 loss 없는지 확인

### S5 process crash
handler 성공 직후 crash -> 재처리 허용

### S6 lease 경쟁
두 러너가 동시에 시작 -> 하나만 commit

### S7 schema drift
추가 컬럼 생겨도 기본 dict payload 유지

### S8 empty tick
변경 없음 -> no-op 성공

### S9 backfill
대량 row를 여러 tick에 걸쳐 안정적으로 소화

## 4. 비기능 테스트

### 성능
- 1k / 10k / 100k row catch-up
- batch size 별 latency/throughput

### 내구성
- 24시간 soak test
- random restart
- storage transient fault injection

### 호환성
- Python 3.11 / 3.12
- SQLAlchemy minor versions
- Functions Core Tools 4.x

## 5. CI 구성

추천 matrix:
- OS: ubuntu-latest
- Python: 3.11, 3.12
- DB: postgres, mysql, mssql

Stage:
1. lint
2. unit
3. integration
4. e2e nightly

## 6. 성공 판정

MVP 기준:
- loss 0
- known duplicate windows만 존재
- unsupported delete는 문서대로 동작
- checkpoint recovery deterministic
