# 아키텍처

## 1. 상위 구조

`azure-functions-db`는 다음 3개 상위 모듈로 구성된다.

1. **core**
   - `DbConfig`
   - `EngineProvider`
   - shared types
   - shared errors
   - serializers
2. **trigger**
   - 기존 polling orchestrator, source adapter, state 관리, Azure Functions 연동
   - `core`를 소비해 pseudo trigger 실행을 담당
3. **binding**
   - `DbReader`(input binding)
   - `DbWriter`(output binding)
   - invocation 단위 session/connection lifecycle 관리

### 1.1 core 레이어

공통 DB 설정/엔진/타입/에러/직렬화 책임을 담당한다.

- `DbConfig`: DB URL, schema, table 등 공유 설정 표현
- `EngineProvider`: config 기준 lazy singleton engine/pool 관리
- shared types: `Row`, `RowDict`, 공통 payload type
- shared errors: trigger/binding이 공통으로 상속하는 기반 예외
- serializers: DB value normalization, row dict 변환

### 1.2 trigger 레이어

기존 trigger 관련 계층은 유지하되, DB 연결/타입/에러/직렬화는 `core`를 사용한다.

### 1.3 binding 레이어

binding 레이어는 Azure Functions 함수 본문에서 직접 호출하는 imperative API를 제공한다.

- `DbReader`: PK 기반 단건 조회, raw SQL 조회
- `DbWriter`: insert/upsert/update/delete 및 batch write
- per-invocation session management: 함수 호출 단위로 session을 열고 닫는 기본 정책

### 1.4 의존성 규칙

- `trigger -> core` 의존만 허용
- `binding -> core` 의존만 허용
- `trigger <-> binding` 상호 import 금지

### 1.5 binding 실행 흐름

```text
Azure Function Invocation
    -> user function
        -> DbReader / DbWriter 생성
            -> DbConfig 구성
            -> EngineProvider.get_or_create(config)
            -> invocation session open
            -> read / write 실행
            -> success: 결과 반환 또는 commit
            -> failure: 예외 surface
            -> close()
```

## 2. 실행 흐름

```text
Azure Timer Trigger
    -> PollRunner.start()
        -> StateStore.load()
        -> StateStore.acquire_lease()    # CAS with ETag
        -> SourceAdapter.fetch_changes()
        -> core normalizer(raw records → RowChange)
        -> Handler.invoke(events)
        -> StateStore.commit()           # checkpoint + lease in one CAS write
        -> StateStore.heartbeat()        # if handler is long-running
```

## 3. 핵심 컴포넌트

### 3.1 PollRunner
전체 lifecycle coordinator.

책임:
- tick 단위 실행
- overlap 방지
- batch 단위 commit
- handler 예외 surface
- retry 친화적 예외 분류

### 3.2 SourceAdapter
각 DB에서 변경분을 읽어 오는 추상화.

핵심 메서드:
- `validate()`
- `open()` / `close()`
- `fetch_changes(checkpoint, limit)` → `FetchResult` (raw records + next_checkpoint)
- `capability()` → `AdapterCapability`

adapter는 raw record를 반환하고, core가 RowChange로 정규화한다.

### 3.3 StateStore
checkpoint와 lease를 **단일 state blob**으로 관리한다.

포함 정보:
- current watermark + tiebreaker PK
- last successful batch metadata
- lease owner / fencing_token / expires_at / heartbeat_at
- source fingerprint

모든 상태 변경은 **ETag 기반 CAS(conditional write)**로 수행되어
lease 획득, heartbeat, checkpoint commit이 원자적으로 이루어진다.

### 3.4 EventNormalizer (core 내부)
adapter가 반환한 raw record를 공통 `RowChange` 이벤트로 정규화한다.
이 로직은 core에 속하며, adapter는 raw record만 반환한다.

### 3.5 HandlerInvoker
사용자 handler를 다음 두 방식으로 호출한다.

- simple mode: `handler(events)`
- rich mode: `handler(events, context)`

## 4. MVP 전략: cursor polling

기본 알고리즘:

```text
checkpoint = (cursor_value, tiebreaker_pk)

SELECT ...
FROM table
WHERE (cursor > :cursor_value)
   OR (cursor = :cursor_value AND pk > :tiebreaker_pk)
ORDER BY cursor ASC, pk ASC
LIMIT :batch_size
```

### 이 전략의 장점
- DB 범용성 높음
- SQLAlchemy로 통합하기 좋음
- 로컬/운영 재현 가능
- 디버깅 쉬움

### 한계
- hard delete 감지 불가(대체 전략 필요)
- updated_at 정밀도와 clock skew에 민감
- row-level old/new diff 제공 어려움
- true event stream은 아님

## 5. 대체 전략

### 5.1 OutboxStrategy
서비스가 outbox 테이블을 기록하고, 라이브러리는 outbox만 소비.

### 5.2 HashDiffStrategy
주기적으로 snapshot/hash를 비교해 변경 추론.
비용이 커서 특수 케이스용.

### 5.3 NativeChangeStreamStrategy
DB가 자체 change stream/CDC를 제공할 때 사용.

예:
- PostgreSQL logical decoding
- MySQL binlog 계열
- MongoDB change stream
- SQL Server change tracking / CDC

## 6. Lease 설계

Timer trigger가 실행되더라도 scale-out 또는 restart 타이밍에 overlap 위험이 있다.
따라서 라이브러리는 **단일 state blob 내에 lease를 포함**하여 관리한다.

state blob 내 lease 필드:
- owner_id
- fencing_token
- acquired_at
- expires_at
- heartbeat_at

원칙:
- 유효한 lease가 있으면 다른 인스턴스는 skip
- handler 시간이 길면 heartbeat 필요
- fencing token이 낮은 writer는 commit 금지
- lease 획득/갱신/commit 모두 **ETag CAS**로 원자적 수행
- 별도 lease blob 없음 — TOCTOU race 방지

## 7. Commit 모델

기본값은 **batch success 후 checkpoint advance**.

```text
fetch batch -> invoke handler -> success -> commit new checkpoint
                               -> failure -> no checkpoint advance
```

이 모델은 중복은 허용하지만 누락을 줄이는 방향이다.

## 8. 파티셔닝

MVP는 단일 logical stream 기준.
후속 버전에서 partitioning 도입.

partition key 후보:
- schema/table
- tenant_id
- modulo hash(pk)
- native partition id(change stream 기반)

## 10. 아키텍처 원칙

- 실행은 단순하게, semantics는 명확하게
- DB reach는 넓게, guarantee는 보수적으로
- handler는 순수 함수처럼, orchestration은 프레임워크가 담당
- 운영은 기본 내장, 추상화는 얇게
