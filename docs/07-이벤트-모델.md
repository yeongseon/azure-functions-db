# 이벤트 모델

## 1. 목표

DB별로 제각각인 결과를 handler가 공통 방식으로 소비할 수 있게 한다.

## 2. 표준 이벤트 객체

```python
@dataclass
class RowChange:
    event_id: str
    op: str
    source: SourceDescriptor
    cursor: CursorValue
    pk: dict[str, object]
    before: dict[str, object] | None
    after: dict[str, object] | None
    metadata: dict[str, object]
```

## 2.1 정규화 책임 분리

adapter는 `RawRecord`(row dict + cursor + pk)를 반환하고,
core의 `EventNormalizer`가 이를 `RowChange`로 변환한다.

정규화 과정:
1. `RawRecord.row` → `after` dict (cursor polling에서는 before=None)
2. `source_descriptor` + `cursor` + `pk` → `event_id` 생성
3. adapter capability에 따라 `op` 결정
4. datetime/Decimal/UUID 등 타입 정규화

이 분리를 통해 adapter는 DB 접근에만 집중하고,
event 구조와 식별자 생성 로직은 core에서 일관되게 관리한다.

## 3. 필드 의미

### event_id
중복 제거용 stable identifier.

기본 생성 규칙:
```text
sha256(
  poller_name +
  source_fingerprint +
  op +
  normalized_cursor +
  normalized_pk
)
```

### op
- `insert` — 새 row 삽입 (CDC/outbox 전략에서 확정 가능)
- `update` — 기존 row 변경 (CDC 전략에서 확정 가능)
- `upsert` — insert 또는 update이나 구분 불가 (cursor polling 기본값)
- `delete` — row 삭제 (CDC/soft-delete 전략에서만)
- `unknown` — 판별 불가

MVP cursor polling은 대부분 `upsert`를 사용한다.
adapter capability에 `can_distinguish_insert_update: bool`이 있어야
`insert`/`update`를 분리 발행할 수 있다.

### source
```json
{
  "kind": "sqlalchemy",
  "dialect": "postgresql",
  "driver": "psycopg",
  "database": "orders",
  "schema": "public",
  "object": "orders"
}
```

### cursor
```json
{
  "kind": "timestamp+pk",
  "value": "2026-04-07T01:23:45.123456Z",
  "tiebreaker": {"id": 12093}
}
```

### pk
원본 row의 primary key dict.

### before / after
- polling 전략에서는 `before=None`인 경우가 많음
- native CDC 전략은 `before` 지원 가능
- delete면 `after=None` 가능

### metadata
예:
- batch_id
- fetched_at
- source_query
- raw_resume_token
- attempt
- partition_id

## 4. 직렬화 규칙

이벤트는 JSON-serializable 해야 한다.

정규화:
- datetime -> RFC3339 UTC string
- Decimal -> string
- UUID -> string
- bytes -> base64
- enum -> string

## 5. Payload 크기 정책

MVP는 full row payload 전달을 기본값으로 한다.
향후 옵션:
- projected columns
- key-only mode
- custom serializer

## 6. Pydantic mapping

선택 기능:

```python
class OrderChange(BaseModel):
    id: int
    status: str
    updated_at: datetime

@db.poll(..., model=OrderChange)
def handle(events: list[OrderChange], context):
    ...
```

주의:
- model validation failure는 기본적으로 batch failure
- 향후 `drop_invalid=True` 옵션 검토

## 7. Downstream 전파 규칙

Service Bus/Event Hub relay 시 envelope:

```json
{
  "specversion": "azure-functions-db/1.0",
  "type": "db.row.change",
  "id": "evt_...",
  "time": "2026-04-07T01:23:46Z",
  "source": "azure-functions-db://orders",
  "data": {
    "...": "RowChange payload"
  }
}
```

CloudEvents 호환은 후속 옵션.
