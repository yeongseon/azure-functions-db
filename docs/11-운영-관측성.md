# 운영 / 관측성

## 1. 목표

문제 발생 시 운영자가 다음 질문에 답할 수 있어야 한다.

- 지금 poller가 돌고 있는가?
- 마지막 성공 시점은 언제인가?
- 얼마나 뒤처졌는가?
- 실패 원인은 source / handler / commit 중 무엇인가?
- duplicate가 증가하고 있는가?

## 2. 구조화 로그

모든 주요 단계에 structured log를 남긴다.

필수 필드:
- poller_name
- invocation_id
- batch_id
- source
- schedule_time
- fetched_count
- committed
- checkpoint_before
- checkpoint_after
- lease_owner
- fencing_token
- duration_ms
- error_type

## 3. 메트릭

### Counter
- `azfdb_batches_total`

### Histogram

- `azfdb_handler_duration_ms`

### Gauge
- `azfdb_lag_seconds`
- `azfdb_last_success_timestamp`
- `azfdb_lease_staleness_seconds`

## 4. Trace

가능하면 각 tick을 하나의 trace/span으로 묶는다.

span 구조:
- poll tick
  - lease acquire
  - fetch
  - normalize
  - handler
  - checkpoint commit

## 5. Lag 정의

lag는 전략별로 다르다.

### cursor timestamp 기반
```text
lag_seconds = now_utc - last_successful_cursor_timestamp
```

### integer sequence 기반
DB max(version)와 checkpoint 차이를 사용.

## 6. 알람

최소 알람:
- 15분 이상 성공 없음
- 연속 5회 실패
- lag 급증
- lease stale
- quarantine 증가

## 7. 운영 명령

CLI 또는 관리 스크립트로 지원:

- checkpoint inspect
- checkpoint reset
- checkpoint clone
- force lease break (주의)
- dry-run fetch
- backfill preview

## 8. Quarantine

batch가 반복 실패하면 payload를 quarantine sink로 보낼 수 있다.

MVP:
- optional blob sink

형식:
```json
{
  "poller_name": "orders",
  "batch_id": "batch_...",
  "error": "...",
  "events": [...]
}
```

## 9. 운영 대시보드

권장 카드:
- poller health table
- last success
- lag
- events/min
- failure rate
- top error types

## 10. 런북 예시

### 배치가 계속 실패할 때
1. checkpoint 확인
2. source query dry-run
3. offending row 확인
4. handler code rollback 여부 판단
5. 필요 시 quarantine 후 checkpoint advance 수동 수행 금지 원칙 유지

### lag가 급증할 때
1. batch size 증가 검토
2. max_batches_per_tick 증가
3. schedule 조정
4. handler downstream 병목 확인
